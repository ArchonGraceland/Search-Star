1. Overview

Search Star is an open-source standard for self-sovereign personal data profiles. An individual hosts their profile at a URI they control, served as JSON-LD. Platforms — dating apps, recruiters, investment networks, brands — query the individual's API endpoint and pay per query at a price the individual sets.

The standard covers four data domains: financial standing (expressed as age-cohort percentiles, not raw dollars), personal presence (the Presence Composite: Rizz, Vibe, Drip), professional capability (skills, credentials, interests), and an advertising profile (high-value targeting signals the user explicitly approves). Each field can be independently scoped to one of three access tiers.

Profiles are backed by a validation layer where trusted third parties stake money against the accuracy of profile claims. If claims are proven false, validators forfeit their stakes to refund advertisers.

Search Star inverts the data economy. Instead of platforms harvesting user data for free and selling it to advertisers, individuals own, host, price, and monetize their own data directly.

2. Identity & Registration

Each Search Star profile is assigned a permanent, sequential identifier in the format SS-XXXXXX (e.g., SS-000001). This number never changes and is globally unique.

Identity is anchored by a W3C Decentralized Identifier (DID) using the did:web method, resolving to a domain the profile owner controls. KYC verification is required and attested by a third-party provider (e.g., Stripe Identity). The DID, KYC attestation, and profile number together form the identity triple.

3. Profile Schema

Profiles are served as JSON-LD documents at the owner's endpoint. The schema context is https://schema.searchstar.org/v0.3. All fields support field-level access control via the accessTier property.

3.1 Financial (Age-Percentile Model)

All financial metrics are expressed as percentiles within the owner's age cohort (5-year bands). No raw dollar amounts ever leave the profile. This makes a 26-year-old with $100K in savings and a 50-year-old with $2M directly comparable — both might be 90th percentile for their cohort.

Percentile data is sourced from the Federal Reserve Survey of Consumer Finances and verified via zero-knowledge proofs (ZK-SNARKs). The proof attests that the owner's actual value meets or exceeds the claimed percentile threshold, without revealing the underlying number.

3.2 Presence Composite™

The Presence Composite is a three-dimensional score measuring interpersonal magnetism, aesthetic taste, and visual self-expression. It is deliberately brand-blind — the algorithm has no input for price tags, brand names, or logos.

RizzInterpersonal magnetism & charismaWit, warmth, presence, confidence

DripVisual expression & personal styleFit & silhouette, color sense, originality, context match

Confidence discount

If Presence is scored from self-assessment only (no photos, no peer reviews, no social content), a 0.85 confidence discount is applied to all factor scores. Providing visual evidence (outfit photos, space photos, social content) removes the discount for the relevant dimensions.

Data sources

Presence scores are derived from: photo AI analysis (fit, color, composition), blinded peer reviews (reviewers cannot see others' scores), space photos (home and workspace aesthetic), taste signals (music, film, reading), anonymized conversation samples, and social content originality indexes.

3.3 Skills & Credentials

Skills use the W3C Verifiable Credentials standard. Each skill has a name, self-reported level (beginner through expert), a numeric score (0–100), and an optional verifiable credential with issuer DID and cryptographic proof.

3.5 Advertising Profile New in v0.3

The advertising profile makes a user's data valuable to advertisers without exposing them to exploitation. It contains four sub-sections, each composed of signals the user explicitly approves before publication.

Life stage signals

Broad categorical descriptors of where the user is in life. These are inferred from identity, financial, and interests data — the user does not self-categorize into advertising buckets. Each signal includes an advertiserRelevance field that makes the value proposition explicit to querying platforms.

Example signalAdvertiser relevance
Independent self-directed professionalHigh-agency decision-maker, not reachable through employer channels
High-net-worth, low-debt householdPremium financial services, wealth management
Education-involved parentEdTech, curricula, learning platforms
International cultural travelerPremium travel, cultural tourism
Active investorFintech, alternative investments, advisory

Consumption profile

Category-level spending patterns. Never dollar amounts — only qualitative tiers: low, moderate, high, very high. Each entry includes category, frequency, scope, and estimated annual value tier.

Professional openness

What kind of professional inbound the user welcomes: advisory/board roles, consulting, mentorship, collaboration, speaking/teaching, or none. This is asked directly — it cannot be reliably inferred.

Demographic value tags

Shorthand tags for advertising audience segmentation, generated automatically from the profile and approved by the user. Examples: "High-net-worth male 55+", "Cultural luxury segment", "Tech-literate self-directed", "Education decision-maker".

Privacy guardrails. The advertising profile MUST NOT include: medical conditions or health data (insurance discrimination risk), specific financial targets or transaction intentions (front-running risk), or relationship status details beyond broad life stage. The user approves every signal, category, and tag. Nothing is published without explicit consent.

4.2 Pricing Model

All pricing is per-query (or per-message for Marketing tier), set by the profile owner. There are no subscriptions, no flat rates, and no free tiers. Owners can set different prices per platform. Billing is handled via Stripe Connect with real-time settlement.

The advertising profile section increases private tier value by 2–5x over a profile without it. Higher Presence scores, verified financials, and higher trust scores (see section 5) all justify higher prices.

A message costs what it costs. There are no refunds on Marketing tier messages. The price is the filter — it protects the profile owner's attention and signals seriousness from the sender.

4.3 Marketing Tier Rules

  • Maximum message length: 500 characters
  • Sender must verify identity before sending
  • Profile owner can permanently block any sender
  • No refunds — payment is for the opportunity to reach the owner, not a guarantee of response

5. Validation & Trust Layer New in v0.3

The validation system allows trusted third parties to vouch for the accuracy of a user's profile by staking money against it. This creates a trust signal for advertisers and querying platforms — a profile backed by real financial stakes is more credible than an unvalidated one.

5.1 Validators & Staking

Only people the profile owner explicitly approves can validate. Validators review specific sections they have personal knowledge of and choose how much to stake (minimum $1.00). Stakes are held in escrow via Stripe Connect.

Validator relationship types

RelationshipBest suited to validate
colleagueSkills, professional openness

familyLife stage signals
mentorSkills, professional trajectory
collaboratorSkills, project output
acquaintanceGeneral identity, location
professional_contactIndustry reputation, skills

Validators see only the sections selected for their review, in a read-only validation view. They never see raw financial data (only percentile bands) and never see access policy pricing.

A validator may withdraw at any time. Their stake is returned (minus any penalties from prior upheld disputes), their validation is removed, and the trust score is recalculated.

5.2 Automated Verification

For fields that can be verified programmatically, the system runs automated checks and records results. These do not require a human validator or staking.

5.3 Trust Score

Each profile receives a composite trust score (0–100) based on three inputs:

ComponentWeightScoring
Validator coverage0–40 pointsPercentage of profile sections with at least one human validator
Total stake0–30 pointsLogarithmic: $0 = 0 pts, $10 = 10, $100 = 20, $1,000+ = 30
Automated checks passed0–30 pointsProportion of verifiable fields confirmed by automated sources

Trust score directly influences suggested query pricing. A profile with a trust score of 80+ can justify 3–5x the base query price.

5.4 Dispute Resolution

When information is proven wrong, the following process applies:

  1. Automated check fails. If a previously-passing automated check returns a contradictory result on re-verification, the system flags the field automatically.
  2. Advertiser dispute. A querying platform that paid for profile data can file a dispute with evidence of material inaccuracy.
  3. Review. Automated-source contradictions are upheld automatically. Subjective disputes (e.g., Presence score inflation) are flagged for manual review.
  4. Penalty on upheld dispute: The field is marked as disputed. Validators who vouched for it forfeit their stakes: 70% refunded to advertisers, 20% platform fee, 10% to the disputing party.
  5. Bad-faith dispute penalty. If a dispute is rejected (the profile data was accurate), the disputing party pays a $5 filing fee to deter frivolous disputes.
The profile owner may update any disputed field and request re-verification. The trust score is recalculated after every dispute resolution.

6. Profile Numbering

Profiles are assigned sequential numbers in the format SS-XXXXXX, zero-padded to six digits. The number is permanent — it never changes, even if the owner changes their handle or domain.
