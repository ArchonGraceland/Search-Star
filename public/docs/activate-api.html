Overview

The Activate Discovery API takes minimal identifying information about an individual and returns a draft profile assembled from public sources scattered across the internet. The response is a flat array of SeededField objects (one per discovered claim) and a parallel array of NarrativePhoto objects (one per discovered image), each tagged with provenance metadata so the reviewing user can confirm, correct, or remove anything before publishing.

This is the backend that powers Step 2 of the /activate flow. It is internal to the Search Star application — there is no public API key for it. This documentation exists for contributors working on the Activate codebase and for anyone evaluating the JSON-LD output downstream of the publish step.

Where this fits in the spec. The conceptual design of the discovery system is in Section 3.9 of the Search Star specification. This document describes the as-built v1.3.0 implementation: which sources are wired up, what their request and response shapes look like, how confidence scores are derived, and how provenance states transition through the pipeline.

Endpoint

POST/api/activate/discover

Request body

{
  "fullName": "Jane Smith",            // required
  "employer": "Anthropic",             // optional but improves disambiguation
  "city": "San Francisco",             // optional but improves disambiguation
  "linkedinUrl": "https://...",        // optional, used as a strong hint
  "disambiguationSelections": {        // optional, see "Disambiguation" below
    "github": "janesmith",
    "linkedin": "jane-smith-123"
  }
}

Response (200 OK)

{
  "fields": SeededField[],
  "photos": NarrativePhoto[],
  "sources": [
    { "name": "GitHub",                  "status": "found",     "count": 12 },
    { "name": "Google Scholar",          "status": "not_found", "count": 0  },
    { "name": "LinkedIn",                "status": "found",     "count": 5  },
    { "name": "Professional Directories","status": "not_found", "count": 0  },
    { "name": "Athletic Records",        "status": "found",     "count": 3  },
    { "name": "Social & Community",      "status": "not_found", "count": 0  }
  ],
  "errors": ["Athletic discovery failed: timeout"],   // present only if any module errored
  "disambiguation": [DisambiguationCandidate],        // present only if multiple matches
  "profileId": "uuid"                                 // present if a directory stub was created
}

If the Supabase write fails, profileId will be omitted but the discovered fields are still returned in the response — the discovery API is best-effort on persistence and never blocks the user-visible result on a database error.

Core types

SeededField

One row per discovered claim. The same logical field (e.g. "Skills · Python") can appear multiple times if more than one source produced a value — this is the multi-value preservation rule from the spec. The reviewing user resolves the conflict by confirming one and removing the others, or by editing one into a corrected value.

interface SeededField {
  id: string                  // stable identifier for the row
  section: string             // 'Identity' | 'Skills' | 'Interests (athletic)' |
                              // 'Interests (social)' | 'Interests (intellectual)' |
                              // 'Professional history' | 'Skills & credentials'
  label: string               // 'Python', 'Marathon time', 'Current title', ...
  value: string               // The discovered value, ready to display
  source: string              // 'github.com', 'scholar.google.com', 'linkedin.com', ...
  sourceUrl: string           // direct link to the source page
  provenance: 'seeded'        // always 'seeded' on first discovery; client mutates it
                              // to 'confirmed' / 'corrected' / 'self_reported' / 'removed'
                              // as the user reviews
  discoveredAt?: string       // ISO 8601 timestamp of when the source was scraped
}

NarrativePhoto

One row per discovered image. Returned alongside fields from the same /api/activate/discover call — Phase 9 photo discovery runs as part of the same pipeline. All photos require explicit user approval in Step 5 of the Activate UI; nothing is auto-included in the published profile.

interface NarrativePhoto {
  id: string
  chapter: 'intellectual' | 'social' | 'athletic'
         | 'professional' | 'aesthetic' | 'family'
  caption: string             // human-readable caption inferred from source context
  date: string                // ISO 8601 if extractable, else empty string
  location: string            // city/venue if extractable, else empty string
  source: 'public'            // for discovered photos; 'google' / 'upload' / 'url'
                              // for the other photo channels
  sourceLabel: string         // 'Conference speaker pages', 'University faculty pages', ...
  previewUrl: string          // direct image URL for thumbnail rendering
  relatedFields: string[]     // array of profile field paths the photo illustrates
                              // e.g. ['skills.python', 'interests.intellectual.conference_speaking']
  // Discovered photos additionally include:
  sourceUrl?: string          // page the image was found on
  sourceContext?: string      // surrounding text snippet, used for caption inference
}

DisambiguationCandidate

When more than one credible match is found in a single source (commonly LinkedIn or GitHub for common names), the API returns up to a handful of candidates and lets the client present them for the user to pick from. The client then re-calls the endpoint with disambiguationSelections set to lock in the choice.

interface DisambiguationCandidate {
  id: string                  // stable identifier within the source
  source: string              // 'github' | 'linkedin'
  name: string                // display name as it appears on the source
  title?: string              // current title if known
  location?: string
  employer?: string
  url: string                 // canonical URL of the candidate's profile
  avatar?: string             // avatar URL if available
  confidence: number          // 0..1 — how confident the matcher is
  snippet?: string            // a short bio or tagline for context
}

Discovery sources

Six modules run in parallel inside a single Promise.allSettled. Per-source failures surface in the errors array but never block other modules. The sources array in the response always contains exactly six entries (one per module) plus an additional entry for each photo source category that returned results, so the client can show progress regardless of which modules succeeded.

ModuleMethodConfidenceNotes
GitHubGitHub REST API v30.90Searches users by name, scores candidates by name + employer + city overlap, fetches each candidate's full profile and top 100 owned repos. Requires GITHUB_PAT env var to avoid the unauthenticated 60-req/hour limit. Output: per-language Skills rows with proficiency derived from repo count and total stars.
Google ScholarSerpAPI google_scholar_author0.85Returns h-index, total citations, top publications, and fields of study. Mapped to Interests (intellectual).
LinkedInSerpAPI Google search restricted to site:linkedin.com/in/0.70Returns Identity and Professional history fields. Triggers disambiguation when multiple matches return for a common name. Lower confidence than GitHub because the underlying data is scraped from search snippets rather than a structured API.
Professional DirectoriesTargeted lookups against CalBar, NYSED, AICPA, NPI0.80Returns Skills & credentials rows. Each directory has its own scraper; failures are isolated per-directory.
Athletic RecordsAthlinks and RunSignUp public race result pages0.75Returns Interests (athletic) rows including event names, dates, finishing times.
Social & CommunityMeetup public profiles, nonprofit board listings0.65Returns Interests (social) rows. Lowest confidence of the six modules — many false positives on common names.

User-supplied identity fields (the request body's fullName, employer, city, linkedinUrl) are also written to the response as SeededField rows under the Identity section, with source: "user-input" and confidence 0.5. This guarantees the reviewing user always sees the inputs they just typed alongside whatever was discovered.

Confidence scores

Each SeededField is persisted to the profile_fields table with a confidence_score in the range 0–1. The number is derived from the source the field came from, not from the field's content. The mapping lives in getSourceConfidence() in src/app/api/activate/discover/route.ts:

SourceScoreWhy
GitHub API0.90Authoritative API with verified ownership
Google Scholar0.85Scholar profiles are claimed by the author
Professional directories (CalBar, NYSED, AICPA, NPI)0.80Government or industry registries with legal weight
Athletic records (Athlinks, RunSignUp)0.75Events verify identity at registration
LinkedIn (via SerpAPI)0.70Self-reported but commonly verified by employers
Meetup & community0.65Self-reported, no verification
Generic web scrape0.60Fallback for unrecognized sources
User input0.50Unverified at the time of entry — boosted to 0.70 once provenance_status becomes self_reported after publish

Clients should display the confidence score as a visual indicator (e.g. a small bar) next to each discovered field so the user can prioritize their review. Scores should be treated as advisory, not gating — the user always gets the final say.

Provenance state machine

Every field that survives discovery flows through the profile_fields.provenance_status column. The five spec states plus the operational removed state form a directed graph:

seeded ─┬─→ confirmed     (user clicks "looks right")
        ├─→ corrected     (user edits the value; original is preserved)
        └─→ removed       (user deletes the row; soft-delete, not hard-delete)

self_reported              (user adds a brand-new field that no source produced)
        ├─→ corrected
        └─→ removed

validated                  (set by the trust system in a later phase — Activate
                            never writes this state directly)

The five canonical states from Section 3.9 are seeded, confirmed, self_reported, corrected, and validated. The implementation also recognizes removed as an operational sixth state used to filter rows out of the published JSON-LD without losing them from the database — this lets the user undo a deletion before publishing. removed rows are excluded from the final JSON-LD by generateProfileJson().

Photo provenance is separate

Photos use a smaller three-state machine in photo_metadata.provenance_status:

  • discovered — Phase 9 public discovery wrote the row, user has not approved
  • confirmed — user approved a discovered photo
  • self_reported — user uploaded directly via Google Photos Picker, device upload, or URL import

Disambiguation flow

When multiple credible matches are found for a single source (most common with LinkedIn for common names like "John Smith"), the response includes a disambiguation array. The recommended client flow:

  1. First call: POST /api/activate/discover with the user's input.
  2. If disambiguation is non-empty, render the candidates and let the user pick one per source.
  3. Second call: re-issue POST /api/activate/discover with the same body plus disambiguationSelections: { github: "janesmith", linkedin: "jane-smith-123" }.
  4. The discovery engine respects the locked-in candidates and skips the disambiguation step.
Don't auto-pick. The client should always show all candidates and let the user choose, even when one has a much higher confidence score than the others. Auto-picking the "best" match risks attributing someone else's professional record to the user — this is exactly the failure mode that makes scraped profiles untrustworthy in the first place.

Persistence model

The discovery API writes to two tables in addition to returning the response:

directory

If the user is starting a new activation, a row is created with seeding_status: 'unclaimed' and status: 'active'. The handle is derived from the user's name (@first.last). The row will eventually be flipped to seeding_status: 'claimed' when the user completes the publish flow.

profile_fields

Every discovered row is persisted with:

{
  profile_id: uuid,            // FK to directory.id
  section: string,
  label: string,
  value: string,
  provenance_status: 'seeded',
  source_url: string | null,
  source_name: string,         // 'github.com', 'linkedin.com', ...
  seeded_at: timestamptz,
  confidence_score: numeric(3,2),
  sort_order: integer
}

RLS policies on this table prevent writes when the parent directory row has seeding_status = 'unclaimed'. Discovery itself bypasses this constraint by running with the service role; subsequent edits from the Activate UI use the user's session and are blocked until the claim flow flips the row to 'claimed'.

photo_metadata

Photos discovered by Phase 9 are persisted with provenance_status: 'discovered' and source_channel: 'public'. No image bytes are stored — only the URL, the inferred caption, the chapter, and the source context.

Error handling

  • 400 Bad Request — request body missing fullName
  • 500 Internal Server Error — unhandled exception in the discovery engine. The user should be told to retry; the response body includes a generic message and the error is logged server-side.
  • 200 OK with errors array — at least one source module failed but others succeeded. The client should still render the successful results and surface the per-source errors as warnings.

Per-source failures never bubble up as 500s. The pipeline is designed to degrade gracefully — if Athlinks is down but GitHub returns data, the user still sees their GitHub-derived skills.

Schema compatibility downstream

The fields and photos returned by this API are persisted to profile_fields and photo_metadata respectively. When the user reaches Step 6 and clicks Publish, src/lib/activate/generate-profile-json.ts reads back the active rows (excluding removed) and emits a JSON-LD document keyed @context: "https://schema.searchstar.org/v1.3". The publish route (POST /api/activate/publish) validates the document, creates a directory entry with status: 'active', seeding_status: 'unclaimed', and returns the assigned profile number.

Platform queries against the resulting profile go through POST /api/platform/query, which filters on status = 'active' and rejects unclaimed rows with a 403. Marketing sends through POST /api/platform/send-marketing apply the same check. The JSON-LD output is the contract between the discovery pipeline, the registration desk, and the platform query system — all three agree on the v1.3 schema.
